# 開発ワークフロー Phase A：プロジェクトの立ち上げ

[< ガイド目次に戻る](README.md) | [Phase B：日常の開発サイクル >](06-workflow-daily.md)

---

Claude Codeでの開発は「考えてから作る」のが鉄則です。ここではプロジェクトをゼロから立ち上げるワークフローを紹介します。

新しいプロジェクトを始めるとき、いきなりコードを書いても迷子になります。まずは**Claude Codeが賢く動くための「土台」**を順番に整えていきます。

## 全体の流れと依存関係

```
Phase A：プロジェクトの立ち上げ

  A1. サービス概要ドキュメント作成
      │  「何を作るか」を言語化する
      ▼
  A2. /consult スキル作成
      │  A1のドキュメントを読み込む相談相手を作る
      ▼
  A3. /consult で要件を壁打ち
      │  機能一覧、MVP、優先順位を固める
      ▼
  A4. アーキテクチャ決定                ← ここが分岐点
      │  技術スタック、設計パターン、       開発者エージェントの
      │  ディレクトリ構成を決める           中身はこれに依存する
      ▼
  A5. CLAUDE.md 作成
      │  プロジェクトルール、構成、規約を定義
      ▼
  A6. サブエージェント作成
      │  アーキテクチャに合わせた専門家チームを編成
      ▼
  A7. その他のスキル・Hooks 作成
      │  /commit, /review 等の開発用スキル
      ▼
  ─────────────────────────────
  Phase B：日常の開発サイクルへ
```

> **ポイント**: A4（アーキテクチャ決定）が先でないと、A6（サブエージェント）は作れません。
> 「バックエンドはExpress + Prisma」と決まって初めて「Prisma ORMの専門家」というエージェントが定義できます。

---

## A1：サービス概要ドキュメントを作る

プロジェクトのルートに、**サービスの全体像を書いたMarkdownファイル**を作ります。これが全ての起点になります。

書くべき内容：

```markdown
# サービス概要

## サービス名
（仮でOK）

## 何を作るのか（一言で）
例: 飲食店の予約とクチコミを管理するプラットフォーム

## 大切にしたいこと
- ユーザー体験を最優先（シンプルで直感的）
- セキュリティ（個人情報を扱うため）
- スモールスタートで早く市場に出す

## 最終的な目標（ビジョン）
例: 世界中どこでも「困ったらすぐ助けが来る」安心感を提供する

## MVP（最低限必要な機能）
1. ユーザー登録・ログイン
2. プロフィール作成
3. マッチング機能
4. チャット機能
5. 決済機能

## 将来的に追加したい機能
- ビデオ通話
- レビュー・評価システム
- AI推薦
```

この段階ではClaudeと会話する必要すらありません。**自分の頭の中にあるイメージを書き出す**だけでOKです。

---

## A2：/consult スキルを作る

A1で作ったドキュメントを「常に読み込んでから回答する相談相手」を作ります。

```
.claude/skills/consult/SKILL.md
```

```markdown
---
name: consult
description: サービスの要件・企画レベルの相談を行う
---

# /consult - サービス要件相談

## 前提知識の読み込み

以下のドキュメントを必ず読み込んでから回答してください。

1. `docs/service-overview.md` - サービス概要

## 相談相手としての振る舞い

- 対等な立場で議論する（上から目線にならない）
- 良いアイデアは素直に認める
- 懸念点があれば率直に伝える
- 「それはいいですね」で終わらず、深掘りする

## サービスの核心（常に意識する）

> （A1のドキュメントからビジョンや大切にしたいことを転記）
```

> **なぜスキルにするのか？**
> 普通にClaudeに相談しても壁打ちはできます。ただし、会話が長くなってコンテキストをクリアしたり、別のセッションで相談を再開すると、サービスの前提知識がリセットされます。スキルにしておけば**毎回必ずサービス概要を読み込んでから回答**してくれるため、一貫した品質の議論ができます。

---

## A3：/consult で要件を壁打ちする

スキルができたら、ひたすら壁打ちします。

```
あなた: /consult
Claude: （サービス概要を読み込み）準備できました。どんな相談ですか？

あなた: MVPの機能一覧を見てほしい。過不足ないかな？

Claude: MVPとしてはよくまとまっていますが、いくつか気になる点があります。
        - 決済機能はMVPに必要ですか？最初は無料で検証して、
          PMF確認後に追加する手もあります
        - 通知機能がないですが、マッチング後にどう通知しますか？
        - ...

あなた: たしかに。決済はPhase 2にして、プッシュ通知を追加しよう

Claude: いいですね。MVPを整理すると...
```

壁打ちで固まった内容は `docs/service-overview.md` に反映していきます。consultスキルも参照ドキュメントが充実するほど賢くなります。

> **重要：壁打ちの結果は必ずドキュメントに残す**
> consultの会話はセッションが終わると消えます。議論がまとまったら「今の内容を `docs/service-overview.md` に反映して」と指示して永続化しましょう。ドキュメントに残さない限り、次のセッションではゼロから説明し直すことになります。これはPhase Bの日常開発でも同様で、「計画→仕様書として保存」が基本の動きになります（[Step 2.5参照](06-workflow-daily.md#step-25永続化--計画を仕様書としてドキュメントに残す)）。

## A4：アーキテクチャを決定する

要件が固まったら、**技術的な「どう作るか」**を決めます。**ここがサブエージェント作成の前提条件**です。

```
あなた: MVPの要件が固まったので、アーキテクチャを決めたい。
        要件はdocs/service-overview.mdを見て。
        以下の観点で提案してほしい。
        - フロントエンド（Web? モバイル? 両方？）
        - バックエンド（フレームワーク、DB）
        - インフラ（AWS? GCP? Vercel?）
        - 設計パターン（モノリス? マイクロサービス?）
        - ディレクトリ構成
```

Claudeが複数案を提示してくれるので、議論して決定します。決まったら **`docs/architecture.md`** として記録。

```markdown
# アーキテクチャ

## 技術スタック
- Frontend: React Native (Expo) — モバイルアプリ
- Backend: Node.js + Express + TypeScript
- DB: MySQL + Prisma ORM
- Infra: AWS (Lambda, API Gateway, RDS, S3)
- 設計: Clean Architecture + DDD

## ディレクトリ構成
src/
├── domain/        # エンティティ、リポジトリIF、ユースケース
├── infrastructure/ # DB、外部API、ストレージ
├── interface/      # Express routes、controllers
└── ...
```

## A5：CLAUDE.md を作成する

アーキテクチャが決まったら、**Claudeへの指示書**を作ります。

```markdown
# CLAUDE.md

## プロジェクト概要
飲食店の予約とクチコミを管理するプラットフォーム。

## 技術スタック
- Backend: Node.js + Express + TypeScript + Prisma
- Frontend: React Native (Expo)
- DB: MySQL

## 重要なルール
- バックエンドコマンドは必ずDocker内で実行すること
- テストは必ず通してからコミット
- 仕様書は docs/ 配下を参照

## ディレクトリ構成
（A4の内容を転記）

## 参照ドキュメント
- docs/service-overview.md — サービス概要
- docs/architecture.md — アーキテクチャ
```

## A6：サブエージェントを作成する

**アーキテクチャが決まっているから、各エージェントの専門領域を定義できます。**

まずは最小構成の3エージェントから始めます。

| 順序 | エージェント | 理由 |
|------|-------------|------|
| 1 | **system_architect** | 全体設計の番人。他のエージェントが参照するルールの源泉 |
| 2 | **developer** | アーキテクチャに合わせた実装担当（例：`backend_developer`） |
| 3 | **qa_engineer** | テスト作成、品質管理 |

プロジェクトが大きくなって分業が必要になったら、tech_researcher、frontend_developer、ui_ux_designer等を追加していきます。詳しくは[サブエージェント](07-sub-agents.md)を参照してください。

各エージェントに書くべき内容：

```markdown
---
name: backend-developer
description: Express + Prisma でAPIを構築する専門家。（← アーキテクチャに依存）
---

# Role
Node.js (Express), MySQL, Prisma ORM のスペシャリスト。

# Goals
1. docs/ の仕様に従ってAPIを実装する
2. Clean Architectureのパターンを維持する    ← アーキテクチャに依存

# Constraints
- フロントエンドのコードは触らない           ← 分業の定義
- 仕様変更が必要な場合はarchitectに差し戻す  ← architect依存
```

> **なぜアーキテクチャが先なのか？**
> 例えば `backend_developer` の定義に「Prisma ORMの専門家」「Clean Architectureに従う」と書くには、その技術選定が確定している必要があります。アーキテクチャ未定のまま作ると、後で全エージェントを書き直す羽目になります。

## A7：その他のスキル・Hooksを作成する

開発インフラの最後の仕上げ。

```
.claude/
├── skills/
│   ├── consult/SKILL.md     ← A2で作成済み
│   ├── commit/SKILL.md      ← コミットルール
│   └── review/SKILL.md      ← コードレビュー
├── agents/
│   ├── system_architect.md  ← A6で作成済み
│   ├── developer.md         ← 実装担当（例：backend_developer）
│   └── qa_engineer.md       ← 品質管理
└── settings.json            ← Hooks（自動フォーマット等）
```

> 必要に応じて tech_researcher, frontend_developer 等を追加。詳しくは[サブエージェント](07-sub-agents.md)を参照。

**ここまで来たら、開発の土台は完成です。[Phase B](06-workflow-daily.md)へ進みましょう。**
