# クイックデモ：エージェントチームが動くところを見せる

所要時間の目安：**10〜15分**

Phase Aが完了した直後に、エージェントチームの並列開発をデモするための手順です。全機能を実装する必要はなく、**1つの機能をチームで分担する様子を見せる**ことに集中します。

---

## 前提

- Phase A（/kickstart）が完了している
- `docs/specs/` に最低1つの仕様書がある
- `~/.claude/settings.json` にチーム機能が有効化されている

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

---

## デモ前の準備（2分）

### 1. 仕様書を1つ用意する

Phase Aの `/consult` 壁打ちで既に `docs/specs/` に仕様書があればそれを使います。

なければ、以下の内容で手動で作成してもOKです：

```markdown
# 天気予報表示

## 概要
登録済みのおでかけスポットの天気予報をOpen-Meteo APIから取得して、見やすいカードで表示する。

## 入力
- 登録済みスポットの緯度経度（SQLiteから取得）

## 出力
- 天気アイコン、気温、降水確率を含む天気カード
- スポットごとに現在の天気と24時間予報を表示

## 技術的な制約（重要）
- 天気データの取得はOpen-Meteo API（https://api.open-meteo.com/）を使う
- APIキーは不要（Open-Meteoの無料枠を使用）
- APIレスポンスは30分程度キャッシュする（過剰なリクエストを防ぐ）
- スポットデータはSQLiteから取得（第1サイクルで実装済み）

## APIエンドポイント
バックエンドは以下のAPIを呼び出す：
  https://api.open-meteo.com/v1/forecast?latitude={lat}&longitude={lon}&current_weather=true&hourly=temperature_2m,precipitation_probability

## 作業の分担
- バックエンド：Open-Meteo API連携 + キャッシュ
- フロントエンド：天気予報カードUI
- テスト：APIレスポンスのモックを使ったユニットテスト
```

これを `docs/specs/feature-weather.md` として保存します。

---

## デモ本番（10分）

### ステップ1：見せる前に一言

> 「これからClaude Codeに**チームを作らせます**。1台のPCの中で、リーダー・バックエンド担当・フロントエンド担当・テスト担当のAIが**同時に動いて、お互いにやり取りしながら開発を進めます**。人間のチーム開発と同じことが起きるのを見てください。」

### ステップ2：チームに指示する

```
> docs/specs/feature-weather.md の仕様に基づいて、
  チームで並列開発して。仕様の「技術的な制約」を必ず守ること。

  チーム構成：
  - backend: Open-Meteo API連携 + レスポンスキャッシュ
  - frontend: 天気予報カードUI（天気アイコン・気温・降水確率）
  - qa: APIモックを使ったユニットテスト

  backendが完了したらfrontendに型定義を共有して。
  全員完了したらテストを実行して結果を報告して。
```

### ステップ3：見せどころの解説

チームが動き出したら、`Shift + ↓` でメンバーを切り替えながら解説します。

**見せるポイント：**

```
① チーム編成の瞬間
  「いまリーダーがタスクリストを作って、3人のメンバーを起動しました」

② 並列で動いている様子
  「Shift+↓ で切り替えてみます。ほら、バックエンドがAPI連携書いてる間に
   フロントエンドがカードUI作ってる。同時に動いてます」

③ メンバー間のやり取り
  「バックエンドが完成して、リーダーがfrontendに型定義を共有しました。
   人間のSlackでのやり取りと同じことが自動で起きてます」

④ テスト実行
  「全員の実装が終わって、qaがテストを実行しています。
   これも人間のQAと同じ流れです」

⑤ 完了
  「全メンバーの作業が完了して、リーダーが統合しました。
   これが1台のPCの中で起きていました」
```

### ステップ4：結果を見せる

チームの作業が完了したら：

```bash
# 開発サーバーを起動（技術スタックに応じて）
npm run dev  # or yarn dev, etc.
```

ブラウザでスポットを選択して天気予報カードが表示されれば完璧です。

> チームの実装がうまく動かなくても大丈夫です。デモの主目的は**AIチームが協調して動く過程を見せること**であり、完璧な成果物を見せることではありません。

## デモ後の補足（伝えると良いこと）

```
「いま見たのは実験的な機能ですが、AIの開発はこの方向に進んでいます。

 普段の開発では、もう少し安定した "サブエージェント" という仕組みを使います。
 チームのように並列ではなく、1人ずつ指示するペアプロ的なスタイルです。

 まずはサブエージェントから始めて、
 慣れてきたらチーム開発も試してみてください。」
```

---

## うまくいかなかったときの逃げ方

| 状況 | 対処 |
|------|------|
| チームが編成されない | `settings.json` の `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` を確認 |
| メンバーが途中で止まる | リーダーのセッションで「止まってるメンバーに再指示して」と伝える |
| コードがビルドできない | 「デモの主目的はAIチームの協調作業を見せること。コードの完成度は後で詰める」と説明 |
| 時間がかかりすぎる | チーム構成を2人（backend + frontend）に減らしてリトライ |
