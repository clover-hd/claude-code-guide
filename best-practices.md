## 1. 検証手段を必ず与える（最重要）

**公式が「最もレバレッジの高いプラクティス」と明言**しています。Claudeは自分で検証できるとき、劇的に品質が上がります。

```
❌ 「メール認証の関数を実装して」
✅ 「メール認証の関数を実装して。
    テストケース: user@example.com → true, invalid → false, user@.com → false
    実装後にテストを実行して確認して」
```

| 手法 | やり方 |
|------|--------|
| **テスト駆動** | 先にテストを書かせて、それを通す実装を作らせる |
| **スクリーンショット** | UI変更時は「スクショを撮って元と比較して」 |
| **ビルド確認** | 「実装後にビルドが通ることを確認して」 |
| **リンター** | 「eslintが通ることを確認して」 |

検証手段がないと、Claudeは「それっぽいけど動かないコード」を出すことがあります。

## 2. コンテキストウィンドウを管理する

**Claude Codeのパフォーマンスは、コンテキストの埋まり具合に直結します。**これは最も重要なリソースです。

### コンテキストとは

Claudeとの会話、読んだファイル、コマンド出力…全てがコンテキストウィンドウに蓄積されます。満杯に近づくと、Claudeの精度が下がり始めます。

### 管理の鉄則

| コマンド | タイミング | 効果 |
|----------|-----------|------|
| `/clear` | **無関係なタスクに移るとき** | コンテキストを完全リセット |
| `/compact` | **コンテキスト70%付近で** | 会話を要約して圧縮（CLAUDE.mdは再読み込みされる） |
| `/compact APIの変更に集中して` | 特定トピックを保持したいとき | 指示付きで要約内容をコントロール |
| `Esc + Esc` → 「Summarize from here」 | 途中から要約したいとき | 選択地点以降だけ圧縮 |

> **Tip**: [ステータスライン](statusline.md)を設定すると、コンテキスト残量が常時表示され、`/compact` のタイミングを逃しません。

### やりがちな失敗

```
❌ ごちゃ混ぜセッション
   タスクA → 関係ない質問 → タスクAに戻る → コンテキストが無駄で一杯

✅ タスクごとにリセット
   タスクA → /clear → タスクB → /clear → タスクC
```

### サブエージェントでコンテキストを守る

調査作業をサブエージェントに委任すると、大量のファイル読み込みがメインのコンテキストを汚染しません。

```
あなた: サブエージェントを使って、認証システムのトークンリフレッシュの
        仕組みと、既存のOAuthユーティリティがあるか調べて

Claude: （サブエージェントが別コンテキストで調査 → 結果だけ報告）
```

## 3. `/init` でCLAUDE.mdを自動生成する

新しいプロジェクトでは `/init` コマンドを実行すると、Claudeがコードベースを分析して**CLAUDE.mdを自動生成**してくれます。

```bash
cd my-project
claude
> /init
```

ビルドシステム、テストフレームワーク、コード規約を検出して、たたき台を作ってくれます。そこから手動で磨いていきましょう。

---

## 4. CLAUDE.mdは短く、具体的に

CLAUDE.mdは**200行以下**を目安に。長すぎると重要なルールが埋もれてClaudeが無視し始めます。

| 書くべきもの | 書かないもの |
|-------------|-------------|
| Claudeが推測できないコマンド | コードを読めばわかること |
| デフォルトと異なるコードスタイル | 言語の標準的な慣習 |
| テストの実行方法 | 詳細なAPIドキュメント（リンクで十分） |
| ブランチ命名規則、PR規約 | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ判断 | 「きれいなコードを書け」等の自明な指示 |

**Claudeが指示に従わないとき**: CLAUDE.mdが長すぎるのが原因なことが多いです。定期的に見直して、不要な行を削りましょう。

## 5. セッション管理を使いこなす

| 操作 | コマンド | 用途 |
|------|----------|------|
| **セッション再開** | `claude --continue` | 前回の続きから作業 |
| **セッション選択** | `claude --resume` | 過去のセッション一覧から選択 |
| **セッション名変更** | `/rename oauth-migration` | 後で見つけやすい名前をつける |
| **巻き戻し** | `Esc + Esc` または `/rewind` | 任意の地点にコード・会話を復元 |

> **Tip**: セッションをgitのブランチのように扱いましょう。作業ストリームごとに別セッションを維持すると、コンテキストが混ざりません。

## 6. 指示は「何を・どこで・どうしたいか」を具体的に

```
❌ 「バグを直して」
✅ 「src/controllers/user.ts の getUser関数で、
    ユーザーが見つからない時に500エラーが返る。
    404を返すように修正して」

❌ 「カレンダーウィジェットを追加して」
✅ 「ホームページの既存ウィジェット（HotDogWidget.phpを参考に）の
    パターンに合わせて、月選択とページネーション付きの
    カレンダーウィジェットを新規作成して」
```

**ファイルは `@` で直接参照**できます：

```
@src/auth/login.ts を見て、セッション管理の仕組みを説明して
```

## 7. Claudeにインタビューしてもらう

大きな機能を作るとき、自分だけでは考慮漏れが発生します。**Claudeに逆質問させる**のが効果的です。

```
あなた: チャット機能を作りたい。詳しくインタビューして。
        技術実装、UI/UX、エッジケース、トレードオフについて
        自分が見落としていそうな難しい部分を深掘りして。
        全て網羅したらSPEC.mdに仕様書を書き出して。

Claude: いくつか質問させてください。
        1. リアルタイム性はどの程度必要？（即時 vs 数秒の遅延OK）
        2. メッセージの永続化は？（履歴を保持する期間）
        3. ファイル添付は必要？
        4. グループチャットは対応する？
        ...
```

仕様書ができたら**新しいセッションで実装**に入ると、クリーンなコンテキストで集中できます。

## 8. 巻き戻し（Rewind）を活用する

Claudeの全てのアクションは**チェックポイント**として記録されています。`Esc + Esc` または `/rewind` で任意の地点に戻れます。

```
会話の復元    → 会話だけ巻き戻し
コードの復元  → ファイル変更だけ巻き戻し
両方の復元    → 会話もコードも巻き戻し
```

「ダメだったら戻せばいい」ので、リスクのある実装も気軽に試せます。チェックポイントはセッションをまたいで保持されます。

> **注意**: チェックポイントはClaude Codeによる変更のみ追跡します。gitの代わりにはなりません。

## 9. よくあるアンチパターンを避ける

| アンチパターン | 症状 | 対策 |
|---------------|------|------|
| **ごちゃ混ぜセッション** | 1つのセッションで無関係なタスクを次々と | `/clear` でタスクごとにリセット |
| **修正ループ** | 同じ問題を2回以上修正指示しても直らない | `/clear` して、学んだことを含めた良い初期プロンプトで再開 |
| **肥大化CLAUDE.md** | ルールを守ってくれない | 200行以下に削減。Claudeが自分でわかることは書かない |
| **検証なき信頼** | 動きそうに見えるが実はエッジケース未対応 | テスト・ビルド・リンター等の検証を必ず指示 |
| **無限調査** | 「調べて」でスコープを絞らず、大量のファイルを読まれる | スコープを絞るか、サブエージェントに委任 |

## 10. Auto Memory（自動メモリ）を活用する

Claudeは会話を通じて学んだことを**自動的にメモ**として保存します。次のセッションでも活きるので、繰り返し同じことを教える必要がありません。

```
あなた: このプロジェクトではpnpmを使って。npmは使わないで。
Claude: 了解しました。（自動でメモリに保存される）

# 次のセッション
あなた: パッケージを追加して
Claude: （メモリからpnpm使用を読み込み済み）
        pnpm add ...
```

| コマンド | 説明 |
|----------|------|
| `/memory` | 現在のメモリファイル一覧を表示・編集 |
| 「これを覚えておいて」 | Claudeに明示的にメモリに保存させる |
| 「忘れて」 | メモリから削除させる |

メモリは `~/.claude/projects/<project>/memory/` にプレーンテキストで保存されるので、いつでも直接編集・削除できます。

## 11. プロジェクト設定は「育てる」もの

CLAUDE.md、エージェント、スキル、Hooksは**一度作って終わりではありません**。

```
使ってみる → Claudeが間違える → ルールを追加/修正 → 精度が上がる
                                      ↑
                              この繰り返しで育つ
```

CLAUDE.mdはgitにコミットして、チームで共同編集しましょう。時間とともに価値が複利的に増えていきます。

## 12. その他の便利機能

| 機能 | 説明 |
|------|------|
| **`/fast`** | 同じモデルで出力速度アップ。簡単な作業に |
| **`claude -p "プロンプト"`** | 非対話モード。CI/スクリプトからClaude Codeを呼べる |
| **画像の貼り付け** | スクリーンショットやデザインをそのまま貼って「これを実装して」 |
| **パイプ入力** | `cat error.log \| claude` でログを直接渡す |
| **並列セッション** | 複数ターミナルでClaude Codeを同時実行。Writer/Reviewerパターンが強力 |
