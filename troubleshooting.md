## よくある問題と対処法

### インストール・起動

#### Claude Codeが起動しない

```bash
# まずバージョン確認
claude --version

# 問題がある場合は診断コマンド
claude doctor

# 再インストール
curl -fsSL https://claude.ai/install.sh | bash
```

#### 「ログインしてください」と言われる

初回起動時はブラウザでAnthropicアカウントにログインする必要があります。ブラウザが自動で開かない場合は、表示されたURLを手動でコピーしてブラウザに貼り付けてください。

#### WSLでブラウザが開かない

WSL環境ではブラウザが自動で開けないことがあります。表示されたURLをWindows側のブラウザにコピーしてログインしてください。

### レート制限（使いすぎ）

#### 「Try again in X hours」と表示される

Proプランの使用量上限に達しました。

```
対処法：
1. 表示された時間だけ待つ（5時間ローリングウィンドウ）
2. 待っている間に仕様書やドキュメントを手動で整理する
3. 次回から以下の節約テクニックを意識する
```

**予防策：**

| テクニック | 効果 |
|-----------|------|
| `/model sonnet` でSonnetを使う | Opusの1/5のトークン消費 |
| `/clear` でタスクごとにリセット | 不要なコンテキスト蓄積を防止 |
| 具体的に指示する | 試行錯誤の回数が減る |
| Plan Modeで先に計画 | 読み取り専用なので実装より消費が少ない |

→ 詳しくは [コスト・トークン管理](cost.md) を参照

### Claudeの出力がおかしい

#### Claudeが嘘をつく（ハルシネーション）

Claudeが「実装しました」と言っているのに、実際にはファイルが変更されていない、またはコードが間違っている場合があります。

```
対処法：
1. 必ずテストを実行させる（「テスト実行して結果を見せて」）
2. ビルドが通るか確認させる
3. 自分の目でもファイルを確認する

予防策：
- 検証手段を必ず指示に含める
- 「実装後にテストを実行して確認して」を習慣にする
```

#### 同じ間違いを繰り返す

2回修正を指示しても直らない場合は、**同じセッションで粘っても改善しません**。

```
対処法：
1. /clear でコンテキストをリセット
2. 学んだこと（「○○は△△が原因だった」）を含めて最初から指示し直す
3. 別のアプローチを提案する（「さっきの方法はうまくいかなかったので、
   代わりに○○を使って」）
```

#### Claudeが指示を無視する

CLAUDE.mdに書いたルールやスタイルガイドを守ってくれない場合。

```
原因（多い順）：
1. CLAUDE.mdが長すぎる（200行超え）→ 重要なルールが埋もれている
2. 指示が曖昧 → より具体的に書き直す
3. コンテキストが一杯 → /compact または /clear

対処法：
- CLAUDE.mdを200行以下に削減
- 最重要ルールをファイルの先頭に書く
- 「CLAUDE.mdのルールを再読み込みして」と明示的に指示する
```

#### コンテキストが一杯になった

画面に「context window is getting full」のような警告が出たとき。

```
対処法：
/compact                           ← 会話を要約して圧縮
/compact APIの変更に集中して        ← 特定トピックを保持して圧縮
/clear                             ← 完全リセット（別タスクに移るとき）
```

→ 詳しくは [実践的なノウハウ > コンテキスト管理](best-practices.md#2-コンテキストウィンドウを管理する) を参照

### ファイル・コードの問題

#### Claudeが間違ったファイルを編集した

```
対処法：
1. Esc + Esc で巻き戻し（チェックポイントに戻る）
2. 「コードの復元」を選択 → ファイル変更だけ元に戻る
3. gitで管理していれば git checkout でも復元可能
```

#### 「Permission denied」でファイルが作れない

```bash
# ファイルのパーミッションを確認
ls -la

# 自分が所有者でない場合
# → 正しいディレクトリで作業しているか確認
# → WSLの場合は /mnt/c/ ではなく ~/projects/ で作業する
```

#### パッケージのインストールで失敗する

```
対処法：
1. エラーメッセージをそのままClaudeに貼る
2. 「このエラーを解決して」と指示する
3. Node.jsのバージョンが古い可能性 → node -v で確認
```

### git関連

#### 「.envをコミットしてしまった」

```bash
# .envをgitの追跡から除外（ファイル自体は残る）
git rm --cached .env

# .gitignoreに追加
echo ".env" >> .gitignore

# コミット
git add .gitignore && git commit -m "fix: .envをgit管理から除外"
```

> **重要**: 一度コミットした認証情報は履歴に残ります。もしAPIキー等を含む.envをpushしてしまった場合は、そのキーを即座に無効化・再発行してください。

#### Claudeが勝手にpushした

通常のClaude Codeの設定では、pushする前に確認が入ります。Auto-Acceptモードでもpushは確認されます。もし意図せずpushされた場合は、承認プロンプトを慎重に読む習慣をつけましょう。

→ [権限モードと承認](overview.md#権限モードと承認重要) を参照

## こんなときどうする？

| 状況 | やること |
|------|---------|
| 何を作ればいいかわからない | Claudeに「初心者でも作れる面白いプロジェクトを5つ提案して」と聞く |
| エラーメッセージが読めない | エラーをそのままコピーしてClaudeに「このエラーを日本語で説明して」 |
| 英語のドキュメントが読めない | Claudeに「このURLの内容を日本語で要約して」 |
| 途中で何をやっていたか忘れた | `claude --continue` で前回のセッションを再開 |
| Claudeの提案が理解できない | 「もっと初心者にわかるように説明して」と遠慮なく言う |
| 全部うまくいかない | `/clear` して最初からやり直す。それでもダメなら別の日に |
