# 実践例：Gitコミット履歴ビジュアライザーを作る

このドキュメントは、[/kickstart スキル](../17-kickstart-skill.md)を使ってプロジェクトをゼロから立ち上げ、Phase Bで開発を進めるまでの実践ガイドです。

実際にこの手順に沿って進めることで、ガイド全体の流れを体験できます。

---

## 事前準備

1. Claude Codeがインストール済みであること（[インストール手順](../01-setup.md)）
2. `/kickstart` スキルがインストール済みであること（[セットアップ手順](../17-kickstart-skill.md#セットアップ)）
3. デモ用リポジトリを用意しておく（下記参照）

### デモ用リポジトリの準備

完成したビジュアライザーで表示するためのリポジトリを事前にcloneしておきます。

```bash
mkdir -p ~/demo-repos && cd ~/demo-repos

# おすすめ：小さくてcloneが速い & コントリビューターが多い
git clone --bare https://github.com/honojs/hono.git
git clone --bare https://github.com/colinhacks/zod.git

# このガイド自体のリポジトリ（「これ自分たちで作ったんだよ」と見せられる）
git clone --bare https://github.com/clover-hd/claude-code-guide.git
```

> `--bare` を付けるとソースコードなしでgit履歴だけ取得するので高速・省容量です。

| リポジトリ | 特徴 | デモでの見どころ |
|-----------|------|-----------------|
| **honojs/hono** | 軽量Webフレームワーク | コントリビューター多数、活発な開発パターン |
| **colinhacks/zod** | TSバリデーションライブラリ | 成長期の急増するコミット頻度 |
| **clover-hd/claude-code-guide** | このガイド自体 | 「AIと一緒に作ったリポジトリ」というストーリー |

### ステップ1（A1）：サービス概要 — Claudeにこう伝える

Claudeが「何を作りたいですか？」と聞いてきます。以下を参考に答えてください。

```
Gitリポジトリのコミット履歴を可視化するWebアプリを作りたい。
```

Claudeが深掘りしてくるので、以下のポイントを伝えてください：

| 質問 | 回答の参考 |
|------|-----------|
| ターゲットは？ | エンジニア個人〜小規模チーム |
| 大切にしたいことは？ | 見た目の楽しさ、すぐ使える手軽さ |
| ビジョンは？ | リポジトリを指定するだけで、チームの開発活動が一目でわかるダッシュボード |
| MVPの機能は？ | 下記参照 |

**MVP機能の例（4つ程度に絞る）：**

1. ローカルのgitリポジトリのパスを指定して読み込める
2. コミット履歴をタイムラインで表示
3. コミッター別の貢献度をグラフ表示
4. 日別・時間帯別のコミット頻度を表示

**将来機能の例：**
- ブランチの分岐・マージの可視化
- ファイル変更のヒートマップ
- GitHub API連携（リモートリポジトリ対応）
- チーム比較ダッシュボード

> **ポイント**: MVPは「最低限これがあれば楽しい」に絞る。多すぎたらClaudeが「後回しにできるものは？」と聞いてくれます。

Claudeが `docs/service-overview.md` を作成したら、内容を確認して「OK」と伝えます。

### ステップ4（A4）：アーキテクチャ決定 ← 最重要

> **ここでOpusに切り替えましょう**。Claudeからも案内があります。

Claudeが技術選択肢を比較表で提示してきます。以下は判断の参考です。

#### 判断のポイント

このプロジェクトは「ローカルで動くWebアプリ」なので：

| 観点 | 推奨の方向 | 理由 |
|------|-----------|------|
| フロントエンド | React or Vue + チャートライブラリ | 可視化がメインなので描画ライブラリの充実度が重要 |
| バックエンド | 軽量フレームワーク | gitコマンドの実行とJSON変換が中心。重厚なものは不要 |
| DB | なし or SQLite | MVPではgitから直接読み取るだけ。キャッシュ用にSQLiteはあり |
| インフラ | ローカル実行 | MVPはlocalhost、将来的にデプロイ可能にしておく |

```
Claudeにこう伝えてもOK：
「チャートライブラリで見た目が映えるものがいい。
 DBはMVPではなしで、gitから直接読み取る方式にしたい。
 技術選定はおまかせで、比較表を見せて。」
```

Claudeが `docs/architecture.md` を作成し、重要な判断は `docs/architecture-decisions/` にADRとして記録します。

## Phase B：開発サイクル

Phase Aが完了したら、以下のサイクルで機能を1つずつ実装していきます。

### サイクルの流れ

```
1. /consult                            → 機能の要件を壁打ち
2. Shift+Tab                           → Plan Modeで実装計画
2.5. 永続化                            → docs/specs/ に仕様書を保存
3. @.claude/agents/xxx.md で実装        → 専門エージェントに指示して実装
3.5. テスト                            → 実装したらテストを実行
4. /commit                             → コミット
```

### サブエージェントの活用

Phase Aのステップ6で作成されたエージェントを、Phase Bの開発サイクルで活用します。

#### エージェントの呼び出し方

Claude Codeのプロンプトで `@` を入力するとファイルパスの補完が効きます。エージェントファイルのパスを指定して呼び出します：

```
> @.claude/agents/backend_developer.md
> （ここに指示を書く）
```

> **補完のコツ**: `@.cl` まで打つと `.claude/` が候補に出ます。そこから `agents/` → ファイル名と選んでいけます。

#### エージェントの使いどころマップ

```
/consult（要件壁打ち）
  ↓
Plan Mode（実装計画）
  ↓ ここで @.claude/agents/system_architect.md に設計レビューを依頼
  ↓
永続化（docs/specs/ に保存）
  ↓
実装 ← ここでエージェントが活躍
  ↓
テスト ← @.claude/agents/qa_engineer.md にテスト作成・実行を依頼
  ↓
/commit
```

#### エージェント間の情報の受け渡し

サブエージェントは現在のコンテキスト（会話の流れ）を参照できます。そのため、情報の受け渡しには2つのパターンがあります。

**パターンA：同じセッションで連続呼び出し（/clear しない）**

```
@.claude/agents/tech_researcher.md で調査
  ↓ コンテキストが残っている
@.claude/agents/backend_developer.md を呼ぶ
  → 前の調査結果が見えるので、そのまま作業を始める
```

- ファイル保存なしでも情報が渡る
- ただしコンテキストを消費する（会話が長くなると精度が下がる）
- `/clear` せず次のエージェントを呼ぶと、前の文脈を見て勝手に動き出すことがある

**パターンB：セッションをリセットしてから呼び出し（/clear する）**

```
@.claude/agents/tech_researcher.md で調査 → docs/research/ に保存
  ↓
/clear でコンテキストをリセット
  ↓
@.claude/agents/backend_developer.md を呼ぶ
  → 「docs/research/xxx.md を読んで実装して」とファイル経由で情報を渡す
```

- ドキュメントとして残るので後から参照できる
- コンテキストが節約できる
- ファイルに保存しないと情報が消える

**推奨はパターンB** です。理由：

1. **再利用性**: 調査結果がファイルに残るので、別のセッションや別のエージェントからも参照できる
2. **コンテキスト節約**: 長い機能開発ではコンテキストの余裕が重要
3. **習慣づけ**: 「成果物は必ず保存する」という習慣は、セッションが消えても知識が残るプロジェクト運営の基本

> パターンAは「ちょっと聞いてすぐ次に渡したい」ような短いタスクには便利です。状況に応じて使い分けてください。

#### 各エージェントの呼び出し例

**tech_researcher — 実装前の調査**

```
> @.claude/agents/tech_researcher.md
> git logの出力をパースしてJSON化するライブラリを調べて。
  自前パースとライブラリ利用のメリデメを比較表にして。
  結果は docs/research/git-log-parsing.md に保存して。
```

```
> @.claude/agents/tech_researcher.md
> コミット頻度のヒートマップを描画するのに適したチャートライブラリを
  3つ比較して。パフォーマンス、カスタマイズ性、バンドルサイズの観点で。
  結果は docs/research/chart-libraries.md に保存して。
```

> 調査系のタスクは tech_researcher に任せるのが効率的です。Read-Onlyで安全ですし、調べものでメインのコンテキストを消費しません。
>
> **重要**: 調査結果は必ず `docs/research/` に保存させてください。サブエージェントのコンテキストはセッション内で閉じるため、保存しないと次のエージェントが参照できません。

**backend_developer — git解析ロジック**

```
> @.claude/agents/backend_developer.md
> docs/research/git-log-parsing.md の調査結果と
  docs/specs/feature-git-reader.md の仕様に従って、
  gitリポジトリからコミット情報を取得するモジュールを実装して。
  ユニットテストも一緒に書いて。
```

**frontend_developer — UI・チャート実装**

```
> @.claude/agents/frontend_developer.md
> docs/specs/feature-timeline.md の仕様に従って、
  タイムラインチャートのコンポーネントを実装して。
  サンプルデータで動作確認できるようにして。
```

**system_architect — 設計レビュー**

```
> @.claude/agents/system_architect.md
> 現在の実装を確認して、以下の観点でレビューして。
  - バックエンドとフロントエンドの責務分離は適切か
  - データの流れに無駄はないか
  - 将来のGitHub API連携を見据えた構造になっているか
```

> 機能が2〜3個揃ったタイミングで一度 system_architect にレビューさせると、早期に設計の歪みを発見できます。

**qa_engineer — テスト**

```
> @.claude/agents/qa_engineer.md
> git解析モジュールのユニットテストを充実させて。
  以下のエッジケースをカバーして：
  - コミットが0件のリポジトリ
  - マージコミットの扱い
  - 日本語を含むコミットメッセージ
```

**technical_writer — ドキュメント**

```
> @.claude/agents/technical_writer.md
> README.md を作成して。以下を含めて：
  - プロジェクト概要とスクリーンショットの配置場所
  - セットアップ手順
  - 使い方
  - 開発方法（コントリビューション向け）
```

#### エージェント活用のコツ

| コツ | 説明 |
|------|------|
| **仕様書を渡す** | `docs/specs/xxx.md の仕様に従って` と指定すると、エージェントが仕様を読んでから実装する |
| **調査結果を保存** | tech_researcher には必ず `docs/research/` に保存させる。保存しないと次のエージェントが参照できない |
| **1エージェント1タスク** | 「UIも作ってAPIも作って」ではなく、担当ごとに分けて指示する |
| **調査→保存→実装の順** | tech_researcher で調査 → `docs/research/` に保存 → 実装エージェントに調査結果と仕様書の両方を渡す |
| **レビューを挟む** | 実装が一段落したら system_architect にレビューさせて品質を保つ |
| **テストは別途** | 実装者にテストも書かせるが、qa_engineer に追加のエッジケーステストを書かせると網羅性が上がる |

## 各サイクルで意識すること

### /consult のあと

- Plan Modeで計画を立てたら、**必ず `docs/specs/` に仕様書を保存**してもらう
- 「仕様書を docs/specs/feature-timeline.md に保存して」と明示的に指示

### 実装中

- 「テストも一緒に書いて」と伝える
- 動くものができたら「テストを実行して結果を見せて」で検証

### コミット

```
> /commit
```

これだけでルールに従ったコミットが作られます。

### 次のサイクルに入る前

```
> /clear
```

コンテキストをリセットしてから次の機能に取り掛かる。

## 困ったときは

| 状況 | 対処 |
|------|------|
| Claudeの実装が動かない | 「テスト実行して」で検証。エラーをそのまま貼って修正指示 |
| 2回直しても直らない | `/clear` して学んだことを含めて最初から指示し直す |
| コンテキストが一杯 | `/compact` で圧縮、または `/clear` でリセット |
| やり直したい | `Esc + Esc` でチェックポイントに巻き戻し |
| 設計に迷ったら | `/model opus` に切り替えて相談 |

## 応用編：エージェントチームで並列開発する

> **これは実験的機能です。** まずは前述のサブエージェント方式でMVPを完成させてから試すことをおすすめします。

### これが何かを一言で

**1台のPC、1つのターミナルの中で、AIがチームを組んで、お互いにコミュニケーションを取りながら並列に開発を進める。**

人間のチーム開発と同じことが、あなたのPC1台で起きます。

### 人間のチーム開発との対比

```
人間のチーム開発                    AIエージェントチーム
─────────────────────────         ─────────────────────────
PM がSlackでタスクを割り振る       → リーダーがタスクリストで割り振る
バックエンド担当が実装する         → backend担当のセッションが実装する
フロントエンド担当が並行で実装する → frontend担当のセッションが並行で実装する
完成したらコードレビュー           → architectがレビューして指摘する
QAがテストする                     → qa担当がテストを書いて実行する
Slackで「マージしていい？」        → メッセージで「実装完了、レビューお願い」
全員が同じGitリポジトリで作業      → 全員が同じGitリポジトリで作業
```

違いは、**これが全部1つのターミナル内で、数分〜数十分で起きる**ということです。

### セットアップ

`~/.claude/settings.json` に以下を追加（未設定の場合）：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### デモ：第2サイクルをチームで並列開発する

第1サイクル（git読み込み）が完成した状態から、第2サイクル（タイムライン表示）をチームで進める例です。

#### 1. リーダーに指示する

```
> 第2サイクルの「タイムラインチャート」をチームで並列開発したい。
  docs/specs/feature-timeline.md の仕様書に基づいて、
  以下のチームを編成して並列に進めて。

  - researcher: チャートライブラリの調査・比較
  - backend: コミットデータの日付グループ化API
  - frontend: タイムラインチャートのUI実装
  - qa: テストの作成
```

#### 2. リーダーがチームを編成して動き出す

リーダーが自動でチームを作り、タスクを割り振ります。以下のようなことが**あなたの目の前で同時に**起きます：

```
┌─────────────────────────────────────────────────────────┐
│ リーダー                                                 │
│ > チームを編成します。タスクリストを作成中...             │
│ > researcher, backend, frontend, qa を起動しました       │
└─────────────────────────────────────────────────────────┘
     │
     ├── researcher が起動
     │   「チャートライブラリを調査中...
     │    Recharts, Chart.js, D3.js を比較しています」
     │
     ├── backend が起動
     │   「コミットデータのグループ化ロジックを実装中...
     │    src/services/commit-grouper.ts を作成」
     │
     ├── frontend が起動
     │   「ダッシュボードレイアウトを実装中...
     │    src/components/Timeline.tsx を作成」
     │
     └── qa が起動
         「テスト戦略を検討中...
          テストケースを洗い出しています」
```

#### 3. チーム内でコミュニケーションが発生する

```
researcher → リーダー:
  「調査完了。Rechartsを推奨します。理由：
   Reactとの親和性が高く、タイムラインチャートの
   サンプルが豊富。比較表を docs/ に保存しました」

リーダー → frontend:
  「ライブラリはRechartsに決定。
   researcher の調査結果を参照して実装を進めて」

backend → リーダー:
  「日付グループ化APIを実装完了。
   テストも通っています」

リーダー → qa:
  「backendの実装が完了した。テストを書いて」

frontend → リーダー:
  「タイムラインコンポーネントの実装完了。
   backendのAPIと繋ぎ込みたいので、
   エンドポイントの仕様を教えて」

リーダー → frontend:
  「backend が作った src/services/commit-grouper.ts を
   参照して。インターフェースはこう...」

qa → リーダー:
  「全テスト作成完了。15件中15件パス。
   エッジケース（空リポジトリ、1コミット）もカバー済み」
```

#### 4. リーダーが統合してコミット

全メンバーの作業が完了すると、リーダーが統合確認してコミットします。

```
リーダー:
  「全メンバーの作業が完了しました。
   - ライブラリ調査 ✓
   - バックエンドAPI ✓
   - フロントエンドUI ✓
   - テスト 15/15 パス ✓
   コミットします。」
```

#### 5. ターミナルでの見え方

`Shift + ↓` でチームメンバーのセッションを切り替えて、**各メンバーが何をしているかリアルタイムで見れます**。

```
┌─ リーダー ──────────────────────────────────────────┐
│ タスクの進捗:                                       │
│ ✓ researcher: ライブラリ調査完了                     │
│ ✓ backend: API実装完了                              │
│ ● frontend: UI実装中... (80%)                       │
│ ✓ qa: テスト作成完了                                │
│                                                     │
│ [Shift+↓ で他のメンバーに切り替え]                   │
└─────────────────────────────────────────────────────┘
```

### もっと大胆に：MVP4機能を一気に並列開発

慣れてきたら、MVP全体をチームに任せることもできます。

```
> docs/specs/ 配下の4つの仕様書に基づいて、
  MVP全機能を並列で開発して。

  チーム構成：
  - backend: git読み込み + データ変換APIの実装
  - frontend-timeline: タイムラインチャート
  - frontend-contrib: コミッター別貢献度グラフ
  - frontend-heatmap: 時間帯別ヒートマップ
  - qa: 各機能のテスト（他メンバーの完了を待って順次実行）

  backendが先に完成したら、frontend系メンバーに共有して。
  全員完了したらqaにテストさせて、最後にまとめてコミット。
```

**人間のチームなら数日かかる作業が、目の前で数十分で進行する** — これがエージェントチームのインパクトです。

### 注意点

| 項目 | 内容 |
|------|------|
| **トークン消費** | メンバー数分のコンテキストを使う。4人チーム ≒ 4セッション分のコスト |
| **実験的機能** | 予期しないエラーやメンバーの停止が起きることがある |
| **仕様書が必須** | チームメンバーは `docs/specs/` の仕様書を読んで動く。仕様が曖昧だと各メンバーがバラバラに実装する |
| **コンフリクト** | 同じファイルを複数メンバーが触るとgitコンフリクトが起きる。担当領域を明確に分けること |
| **最初は小さく** | いきなり全機能ではなく、まずは2〜3人のチームで1機能を試す |

### サブエージェント方式との使い分け

```
サブエージェント（@.claude/agents/xxx.md で呼び出し）
  → 1人ずつ順番に指示する「ペアプログラミング」スタイル
  → 安定している、コスト控えめ
  → まずはこちらから始める

エージェントチーム
  → 複数人が同時に動く「チーム開発」スタイル
  → 並列で速い、ただしコストが高く実験段階
  → サブエージェントに慣れてからステップアップ
```

## まとめ

```
Phase A（/kickstart）で土台を作る          → 約1〜2セッション
Phase B で機能を1つずつ実装                → 1機能 = 1サイクル
  ├─ サブエージェント方式（基本）           → 1人ずつ順番に指示
  └─ エージェントチーム方式（応用）         → AIチームが並列開発
全4機能でMVP完成                           → 数日〜1週間程度
```

大事なのは「一度に全部作ろうとしない」こと。1サイクルごとに動くものを確認しながら進めてください。

エージェントチームは未来の開発スタイルの一端を垣間見せてくれます。まずはサブエージェントで基本を掴み、そこからチーム開発の世界を体験してみてください。
