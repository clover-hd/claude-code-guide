# 開発ワークフロー Phase B：日常の開発サイクル

[< ガイド目次に戻る](README.md) | [< Phase A：プロジェクトの立ち上げ](workflow-setup.md)

---

プロジェクトの土台（CLAUDE.md、エージェント、スキル）が整ったら、日常の開発は以下のサイクルで回します。

## 全体の流れ

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   1. 構想     /consult で要件を壁打ち                   │
│       ↓       「何を作るか」を固める                     │
│                                                         │
│   2. 計画     Plan Mode で実装計画を立てる              │
│       ↓       「どう作るか」を設計する                   │
│                                                         │
│  2.5. 永続化   計画を仕様書としてドキュメントに残す      │
│       ↓       次のセッションでも参照できるようにする     │
│                                                         │
│   3. 実装     計画に沿ってコーディング                  │
│       ↓       小さく作って確認を繰り返す                │
│                                                         │
│  3.5. テスト   ユニットテスト・E2Eテストで検証           │
│       ↓       Claudeが失敗を自動修復するループ          │
│                                                         │
│   4. 記録     /commit でコミット                        │
│               変更を論理単位で記録                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

---

## Step 1：構想 — `/consult` で要件を固める

いきなりコードを書き始めない。まずは**何を作りたいのか**をClaudeと壁打ちします。

```
あなた: /consult
Claude: 準備できました。どんな相談ですか？

あなた: ユーザーのプロフィール画像をアップロードする機能を追加したい

Claude: いくつか確認させてください。
        - 画像のサイズ制限は？
        - トリミング機能は必要？
        - 保存先はS3？ローカル？
        - 既存のプロフィール画面に追加？新しい画面を作る？
```

ここで**機能の範囲・仕様・優先度**を整理します。「作りたいもの」が曖昧なまま実装に入ると、途中で方向転換が発生してコンテキストを無駄に消費します。

> `/consult` スキルがなくても、普通に相談するだけでもOK。
> スキルがあると「毎回サービス仕様を読み込んでから回答」してくれるので精度が上がります。

---

## Step 2：計画 — Plan Mode で設計する

要件が固まったら、**Plan Mode** に入って実装計画を立てます。

### Plan Modeとは

Plan Modeは**読み取り専用の探索・設計モード**です。コードベースを自由に読み回り、実装方針を策定しますが、**ファイルの変更は一切行いません**。

```
できること                     できないこと
✅ ファイルの読み込み           ❌ ファイルの作成・編集・削除
✅ コードベースの検索           ❌ コマンドの実行（npm, git等）
✅ Web検索で最新情報を取得      ❌ パッケージのインストール
✅ サブエージェントで並列調査    ❌ システムに変更を加える操作全般
✅ 質問して要件を明確化
```

### Plan Modeの操作

| 操作 | 方法 |
|------|------|
| **ON/OFFの切り替え** | `Shift + Tab` を押す |
| **起動時にPlanモードで開始** | `claude --permission-mode plan` |
| **計画をエディタで編集** | `Ctrl + G` |

`Shift + Tab` を押すたびにモードが切り替わります：

```
通常モード → Auto-Acceptモード → Plan Mode → 通常モード → ...
              ⏵⏵ accept edits on    ⏸ plan mode on
```

### Plan Modeの使い方（実践例）

```
# 1. Shift+Tab でPlan Modeに入る（画面下部に「⏸ plan mode on」と表示）

あなた: プロフィール画像アップロード機能を実装したい。
        画像はS3に保存、最大5MB、JPEG/PNG対応。
        既存のプロフィール編集画面に追加する。

# 2. Claudeがコードベースを探索し、計画を立てる
Claude: コードベースを確認しました。実装計画を立てます。

        ## 実装計画

        ### 影響ファイル
        - src/domain/entities/UserProfile.ts（画像URLフィールド追加）
        - src/infrastructure/storage/S3Client.ts（新規）
        - src/usecases/UploadProfileImageUseCase.ts（新規）
        - src/interface/routes/profileRoutes.ts（エンドポイント追加）
        - prisma/schema/user.prisma（カラム追加）

        ### ステップ
        1. Prismaスキーマにprofile_image_urlカラムを追加
        2. S3アップロード用のインフラ層を実装
        3. ユースケースを実装（バリデーション含む）
        4. APIエンドポイントを追加
        5. テストを作成・実行

        この計画で進めてよいですか？

# 3. 計画を確認・修正
あなた: リサイズ処理も追加して。アップロード前に800x800にリサイズしたい。

Claude: 計画を修正します。
        ステップ2.5として sharp ライブラリによるリサイズ処理を追加...

# 4. 計画がOKなら、Shift+Tab でPlan Modeを解除
# 5. Claudeが計画に沿って実装を開始
```

### なぜPlan Modeが重要なのか

| Planなし | Planあり |
|---------|---------|
| いきなり実装開始 → 途中で設計ミスに気づく → やり直し | 事前に全体像を把握 → 一発で正しく実装 |
| コンテキストが無駄に消費される | 計画フェーズは読み取り専用で軽量 |
| 「何を変更するか」が不明確 | 影響ファイル一覧が事前にわかる |
| 手戻りのたびにモチベーション低下 | 計画を見ながら着実に進められる |

## Step 2.5：永続化 — 計画を仕様書としてドキュメントに残す

Plan Modeで立てた計画は、あくまで**そのセッション内の会話**です。`/clear` したり、新しいセッションを始めると消えます。だからこそ、**計画が固まったら仕様書として永続化**するのが重要です。

### なぜ永続化が必要なのか

```
セッションA: 計画を立てる → 実装の途中でコンテキストが一杯
                              ↓
セッションB: /clear して再開 → 「あれ、さっきの計画どんなだっけ…」

↓ 仕様書があれば ↓

セッションB: docs/specs/profile-image.md を読み込み → 計画通りに続行
```

Plan Modeの計画はセッションに閉じますが、**ドキュメントはセッションをまたいで生き続けます**。CLAUDE.mdやスキルから参照させれば、Claudeは常にそのドキュメントを踏まえた判断をしてくれます。

### 実践の流れ

```
# Plan Modeで計画が固まったあと、そのまま依頼する

あなた: この計画を docs/specs/profile-image-upload.md に仕様書として保存して。
        以下を含めて：
        - 機能概要
        - 技術仕様（API、DBスキーマ、ストレージ設計）
        - 影響ファイル一覧
        - 実装ステップ

Claude: docs/specs/profile-image-upload.md を作成しました。
```

> **Tip**: Plan Modeは読み取り専用ですが、ドキュメント作成はPlan Modeを解除してから行います。計画を確認→Shift+TabでPlan Mode解除→「今の計画をドキュメントに保存して」と指示する流れです。

### 何をドキュメントに残すべきか

| 残すべきもの | ファイル例 | 理由 |
|-------------|-----------|------|
| **機能仕様** | `docs/specs/feature-name.md` | 「何を作るか」の根拠。consultで練った要件もここ |
| **API設計** | `docs/specs/api/endpoints.md` | エンドポイント、リクエスト/レスポンス形式 |
| **DBスキーマ設計** | `docs/specs/db/schema-changes.md` | テーブル定義、マイグレーション計画 |
| **アーキテクチャ判断** | `docs/architecture-decisions/` | 「なぜこの技術を選んだか」の記録 |
| **実装計画** | `docs/specs/feature-name.md` 内 | ステップ・影響ファイル・注意事項 |

### ドキュメントディレクトリの構成例

```
docs/
├── service-overview.md              ← サービス全体像（Phase A1で作成）
├── architecture.md                  ← アーキテクチャ（Phase A4で作成）
├── specs/                           ← 機能仕様書
│   ├── user-authentication.md
│   ├── profile-image-upload.md      ← ← Plan Modeの計画を永続化
│   ├── chat-messaging.md
│   └── api/
│       └── endpoints.md
└── architecture-decisions/          ← 技術判断の記録
    ├── 001-why-prisma.md
    └── 002-why-clean-architecture.md
```

### CLAUDE.mdとの連携

仕様書の場所をCLAUDE.mdに記載しておけば、Claudeが実装時に自動的に参照します。

```markdown
# CLAUDE.md（抜粋）

## 参照ドキュメント
- 機能仕様: docs/specs/ 配下を参照
- 新機能の実装前に、必ず該当する仕様書を読むこと
```

consultスキルからも参照させると、壁打ちの質が上がります。

```markdown
# .claude/skills/consult/SKILL.md（抜粋）

## 前提知識の読み込み
1. docs/service-overview.md - サービス概要
2. docs/specs/ - 既存の機能仕様（実装済み・計画中のものを把握）
```

### consultで練った要件も永続化する

Step 1（/consult）で議論した要件も、放置すると消えます。壁打ちがいい感じにまとまったら：

```
あなた: 今の議論をまとめて docs/specs/chat-messaging.md に保存して。
        まだ計画段階なので、ステータスを「Draft」にして。

Claude: 作成しました。

        # チャットメッセージ機能
        **ステータス: Draft**

        ## 機能概要
        ユーザー間の1対1リアルタイムチャット...
```

Draft → Plan Modeで設計 → Approved → 実装、という流れで仕様書のステータスも更新していきます。

## Step 3：実装 — 計画に沿ってコーディング

Plan Modeを解除すると、Claudeが立てた計画に沿って実装を進めます。

### 実装中のコツ

**小さなステップで確認しながら進める**

```
あなた: まずステップ1のPrismaスキーマ変更から始めて

Claude: （スキーマを変更 → マイグレーション実行）
        完了しました。次のステップに進みますか？

あなた: OK、次はS3クライアントの実装をお願い

Claude: （S3Client.tsを作成 → テスト実行）
        ...
```

**意図しない方向に進んだら即停止**

```
あなた: （Escキーで中断）
        違う、そのアプローチじゃなくて、multerを使ってほしい
```

**サブエージェントへの自動委任**

大きなタスクの場合、Claudeは自動的にサブエージェントに作業を委任することがあります。例えば：
- バックエンドの実装 → `backend_developer` エージェント
- テストの作成 → `qa_engineer` エージェント
- 技術調査が必要 → `tech_researcher` エージェント

## Step 3.5：テスト — Claudeにテストを書かせて検証する

実装が終わったら、コミットの前にテストです。Claude Codeはユニットテストからブラウザを使ったE2Eテストまで、**テストの作成・実行・失敗の修正を自律的に**行えます。

### ユニットテスト

最も基本的なテスト。関数やモジュール単位で正しく動くか確認します。

```
あなた: UploadProfileImageUseCaseのユニットテストを書いて。
        以下のケースをカバーして：
        - 正常系: JPEG画像のアップロード成功
        - 異常系: 5MB超のファイル → エラー
        - 異常系: サポート外の形式（GIF）→ エラー
        テスト実行して結果を見せて。

Claude: テストを作成しました。実行結果：
        ✅ 3/3 passed
```

> **TDD（テスト駆動開発）のすすめ**：先にテストを書かせて失敗させ、そのテストを通す実装を作る方式が、Claude Codeとの相性が非常に良いです。テストが「正解の定義」になるため、Claudeが自分で検証しながら正確な実装を生み出せます。

```
# TDDの流れ
あなた: まずテストだけ先に書いて。実装はまだしないで。
Claude: テスト作成完了。実行結果：❌ 3/3 failed（期待通り）

あなた: OK、このテストが通るように実装して。
Claude: 実装完了。実行結果：✅ 3/3 passed
```

### E2Eテスト（Playwright）

Playwrightを使えば、**実際のブラウザを操作して**ユーザー目線のテストをClaudeが書いて実行できます。

```
あなた: Playwrightでプロフィール画像アップロードのE2Eテストを書いて。
        1. ログインページでログイン
        2. プロフィール編集ページに遷移
        3. 画像ファイルをアップロード
        4. アップロード成功のメッセージが表示されることを確認
        5. プロフィールページに画像が表示されていることを確認
        テストを実行して結果を見せて。

Claude: E2Eテストを作成しました。

        // tests/e2e/profile-image-upload.spec.ts
        import { test, expect } from '@playwright/test';

        test('プロフィール画像をアップロードできる', async ({ page }) => {
          await page.goto('/login');
          await page.fill('[data-testid="email"]', 'test@example.com');
          await page.fill('[data-testid="password"]', 'password123');
          await page.click('[data-testid="login-button"]');

          await page.goto('/profile/edit');
          await page.setInputFiles('input[type="file"]', 'tests/fixtures/test-image.jpg');
          await expect(page.locator('.upload-success')).toBeVisible();
          ...
        });

        実行結果：✅ 1/1 passed
```

### Claudeのテスト自動修復ループ

Claude Codeの真価は**テストが失敗したときの自動デバッグ**にあります。

```
Claude: テスト実行結果：❌ 1/1 failed

        エラー: locator('.upload-success') を見つけられません

        （スクリーンショットを確認...）
        画面にはアップロード成功のトースト通知が表示されていますが、
        クラス名が .toast-success でした。セレクタを修正します。

        修正後の実行結果：✅ 1/1 passed
```

Playwrightが撮るスクリーンショットやアクセシビリティツリーをClaudeが読み取り、失敗原因を自分で特定して修正する。これが「検証手段を与える」ことの具体的な効果です。

### テスト戦略の全体像

| テストの種類 | ツール例 | 対象 | 速度 |
|-------------|---------|------|------|
| **ユニットテスト** | Jest, Vitest | 関数・モジュール単位 | 速い |
| **インテグレーションテスト** | Jest + Supertest | API エンドポイント | 中程度 |
| **E2Eテスト** | Playwright | ブラウザでの操作フロー全体 | 遅い |

```
         ユニットテスト（多い・速い・安い）
        ████████████████████████████████
       インテグレーションテスト（中程度）
        ██████████████████
        E2Eテスト（少ない・遅い・高い）
        ████████
```

全てをE2Eでカバーする必要はありません。**ユニットテストで基本ロジックをカバーし、E2Eは重要なユーザーフロー**（ログイン、決済、主要機能）に絞るのが定石です。

### E2Eで最低限カバーすべきフロー

- ログイン → ログアウト
- 主要機能の操作（作成・編集・削除）
- 決済・注文（お金が絡む処理）
- 検索・フィルター（利用頻度が高い操作）

## Step 4：記録 — `/commit` でコミット

実装とテストが完了したら `/commit` でルールに従ったコミットを行います。

```
あなた: /commit

Claude: 変更を確認します...

        以下の2つのコミットに分けます：

        1. feat: Prismaスキーマにprofile_image_urlを追加
           → prisma/schema/user.prisma

        2. feat: プロフィール画像アップロードAPI実装
           → src/infrastructure/storage/S3Client.ts
           → src/usecases/UploadProfileImageUseCase.ts
           → src/interface/routes/profileRoutes.ts
           → tests/...

        コミットしてよいですか？
```

## ワークフローの応用パターン

### パターンA：新機能開発（フルコース）

```
/consult → Plan Mode → 実装 → /commit
```

一番丁寧なフロー。新しい機能をゼロから作るとき。

### パターンB：バグ修正（Planから）

```
Plan Mode（原因調査）→ 実装（修正）→ /commit
```

バグの原因がわからないとき。Plan Modeで読み取り専用で調査してから修正。

### パターンC：小さな修正（直接実装）

```
実装 → /commit
```

「テキストの修正」「既知のバグ修正」など、計画不要な小さな変更。

### パターンD：技術調査のみ

```
Plan Mode（調査して終了）
```

「このライブラリ使えるかな？」「この設計で大丈夫？」など、コードを変更しない調査。

## モデルの使い分け

各フェーズでどのモデルを使うべきかは [モデルの違いと選び方](models.md#ワークフロー別の使い分けガイド) にまとめています。基本は **Sonnet 4.6をデフォルトにして、Plan Modeなど深い設計判断が必要な場面だけOpusに切り替える** のがベストです。
